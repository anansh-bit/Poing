1. LEVEL OVERVIEW

Poing is an interactive projection-mapped game that bridges the physical and digital worlds. By leveraging TouchDesigner for real-time visual rendering and Python for complex game logic, the installation transforms a standard physical surface into a dynamic, responsive playground. This document outlines the conceptualization, artistic direction, technical pipeline, and iterative development of the project.

2. CONCEPT & IDEATION

The Core Idea

The initial goal was to create a tangible, communal gaming experience where users interact using their bodies or physical objects rather than traditional controllers.

Inspiration

  • Mario: Platforming aesthetics and goal-oriented level progression.
  • Pinball: Bouncing mechanics, kinetic energy, and physical interactions.
  • Bounce (Nokia): The core loop of collecting items (red stars) to unlock the final goal.

Game Mechanics

  • Objective: Collect red stars scattered across the projection and then navigate the ball into the final goal.
  • Interaction Model: Webcam object tracking makes physical things placed in front of the camera act as solid in-game obstacles for the ball to bounce off of.

3. ART DIRECTION & VISUAL IDENTITY

A major pillar of this project was breaking away from the standard "digital" aesthetic often seen in interactive installations.

THE AESTHETIC

The visual language was strictly defined to evoke a highly tactile, physical feel. All digital assets were designed to match a specific style:

  • Medium: 8-Bit Retro, Pixelated theme.
  • Texture: CRT distortions, Pixles, Gltiches, CRT updating animations.
  • Color Palette: A very red-intensive palette set against a stark white background. This high-contrast approach not only established a bold, artistic identity but also helped the projection stand out brightly against the physical surface, ensuring the tracking and visuals didn't get muddied.

4. TECHNICAL ARCHITECTURE

The backend of the installation relies on a seamless pipeline between physical sensors, TouchDesigner's real-time engine, and custom Python scripts.

Hardware Setup

  • Display: Standard 1024x768 resolution projector.
  • Sensors: 1920x1080 resolution webcam.
  • Compute: My personal laptop running TouchDesigner.

Software Pipeline (TouchDesigner)

The TouchDesigner network was organized into distinct, modular components (COMPs):

  1. Input / Vision Subsystem (TOPs): Capturing the camera feed, applying thresholding, and isolating the player's movement.
  2. Logic Subsystem (CHOPs & DATs): Translating pixel data into coordinate data (X/Y).
  3. Render Subsystem (SOPs & TOPs): Generating the visuals, applying the watercolor and paper texture post-processing effects.
  4. Output / Mapping: Using KantanMapper or Stoner to warp the final output to the physical wall/floor.

5. DEVELOPMENT PROCESS

Phase 1: Sensor Calibration

The first major hurdle was getting reliable input. We started by bringing the camera feed into TouchDesigner and using a Blob Track TOP (or optical flow) to find the user's position.

  • Python Integration: Python scripts within CHOP Execute DATs were used to smooth the tracking data, preventing the interactive elements from "jittering" on screen.

Phase 2: Game Logic with Python

While TouchDesigner is great for visuals, complex state-machines required custom scripting.

  • A Master Text DAT housed the GameManager Python class.
  • Python was used to randomly spawn objects, check for collisions (using math between player X/Y and object X/Y), and update the score variables in TouchDesigner.

Phase 3: Applying the Aesthetic

Bringing the "Red Watercolor on Handmade Paper" look to life required a mix of 2D assets and post-processing in TouchDesigner:

  • Importing pre-painted assets with alpha channels.
  • Using Displace TOPs and Noise TOPs in TouchDesigner to give dynamic, real-time assets slightly wobbly, organic edges mimicking wet paint.
  • Overlaying a high-resolution scan of handmade paper using an Over TOP or Multiply TOP at the very end of the render chain.

Phase 4: Projection Mapping

Once the game was running locally, it was time to map it to the physical world.

  • Made my own calibration file for custom background and deformations using Python, successfully handling the layout without needing external pre-built mapping software.

6. BOSS FIGHTS (CHALLENGES)

  • Challenge 1: Lighting Interference. Ambient light in the exhibition space caused the camera tracking to fail.
    -> Solution: We switched to infrared tracking and used Python to dynamically adjust the exposure based on room brightness.
  • Challenge 2: Python Frame Drops. Running heavy collision loops in Python caused TouchDesigner's framerate to dip below 60fps.
    -> Solution: Optimized the Python code to only check collisions for objects currently on screen, and moved some basic math back into native TouchDesigner CHOPs.
  • Challenge 3: Maintaining the Watercolor Look. Real-time generated particles looked too "digital" against the hand-painted assets.
    -> Solution: Built a custom feedback loop in TouchDesigner to make particles blur and "bleed" into the paper texture.

7. PLAYTESTING

  • Observation: Users initially didn't understand the boundaries of the play area.
  • Refinement: Added a subtle red watercolor border that pulsed when users got too close to the edge of the camera's vision.
  • Feedback: Players loved the tactile feel of the paper texture; it made the wall feel like a living canvas rather than a screen.

8. STAGE CLEAR (CONCLUSION)

Poing successfully proved that interactive installations don't have to look coldly digital. By combining robust Python logic with TouchDesigner's visual power, we created a living, red watercolor painting that responds to human touch.
